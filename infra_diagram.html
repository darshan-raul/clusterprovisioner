④ IAM Roles & Trust Relationships
🛡️

Platform IAM Roles

Three service roles + two cross-account roles in customer accounts

⚡

accio-lambda-role

Principal: lambda.amazonaws.com
  • DynamoDB: PutItem, GetItem, UpdateItem, Query
  • SecretsManager: create/get under accio/users/*
  • KMS: Decrypt, GenerateDataKey
  • StepFunctions: StartExecution
  • STS: AssumeRole → accio-platform-reader (ExternalId)
  • Cognito: AdminUpdateUserAttributes
  • Bedrock: InvokeAgent
🔨

accio-codebuild-role

Principal: codebuild.amazonaws.com
  • CloudWatch Logs: write to /aws/codebuild/*
  • S3: read tf-code, read/write tf-state & outputs
  • KMS: Decrypt, GenerateDataKey
  • StepFunctions: SendTaskSuccess / Failure
  • STS: AssumeRole → accio-platform-provisioner (ExternalId)
  • SSM: GetParameter under /accio/*
🔄

accio-sfn-role

Principal: states.amazonaws.com
  • DynamoDB: UpdateItem, GetItem on clusters table
  • Lambda: InvokeFunction on accio-* functions
  • CloudWatch Logs: log delivery for executions
  • X-Ray: PutTraceSegments, telemetry
Cross-Account Roles in Customer AWS Accounts (deployed via CloudFormation)

🏢 Customer AWS Account

EKS Cluster · VPC · Node Groups
CloudWatch · CloudFormation roles

Onboarding: Customer deploys onboarding_cfn.yaml via a 1-click CloudFormation deep link. This creates accio-platform-reader and accio-platform-provisioner IAM roles with trust policies scoped to the ACCIO platform account ID. Verification is done via GET /onboarding/verify-iam which calls sts:AssumeRole to confirm connectivity.

Encryption & Secrets (ACCIO Account)

KMS + Secrets Manager + SSM

🔑
KMS Master Key
alias/accio-master
Encrypts all S3 buckets + Secrets Manager secrets. 30-day deletion window. Annual key rotation.
🔒
Secrets Manager
accio/users/{sub}/*
Per-user secrets: aws · azure · gcp · github tokens. Created dynamically by Lambda. IAM scoped to user prefix.

SSM Params
/accio/*
Bucket names, KMS ARN. Read by Lambda + CodeBuild at runtime. No hardcoded values.
